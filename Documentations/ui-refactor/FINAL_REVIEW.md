## 3. Architecture summary

Block Lambda Calculus is a block-based simply-typed / Hindley–Milner-style lambda-calculus workbench built on Blockly 12 + TypeScript + webpack. It is **not** a MiniJava editor; no MiniJava terminology exists in production source or tests.

- **Language layer** (untouched by the refactor): `core/blocks` (13 block types), `core/parser/lambdaTextParser.ts`, `core/generator/` (text, editable code, formal derivation), `core/semantics/lambdaReduction.ts` (call-by-structure and call-by-value), `core/machine/csekMachine.ts` (CEK), `core/type-inference/` (HM inference + driver).
- **Renderer**: custom `tude` renderer (`core/renderer/tude.ts`) — Zelos-derived, square reporter shapes, 22×20 value sockets, 16px page gutter, zero corner radius. Registered name `tude` preserved; Zelos and Thrasos remain selectable.
- **Shell**: `core/ui/layout.ts` (panels, resizers, theme, drawers), `core/ui/workbench.ts` (activity/perspectives/palette/shortcuts/diagnostics), `core/ui/visualizationPanel.ts` (bottom panel, traces, lockstep), `core/ui/csekPanel.ts`, `core/renderer/toolbox.ts` (custom toolbox with pointer drag).
- **Styling**: three-layer cascade — `tokens.css` (sole theme authority), `styles.css`, `examples.css`. Grammar colors are installed at runtime onto `documentElement` from the authoritative Blockly theme by `applyLambdaGrammarCssTokens`.
- **Persistence**: `block-lambda-ide-layout-v2`, theme, renderer, active inspector target, autosave workspace/time/interval.

Real, implemented features: lambda abstraction, variables, application, explicit grouping, let, letrec, numeric/boolean literals, numeric/comparison/boolean operators, conditionals, text→block and block→text, editable code, HM type inference with diagnostics, formal typing derivation, program outline, call-by-structure and call-by-value reduction traces, CEK abstract machine with environment and continuation views, lockstep rewrite↔machine correspondence, `.blc` save/load, autosave and recovery, custom renderer, themes, perspectives, command palette, responsive drawers.

## 4. Findings by severity

### Critical

**C1 — `npm run build` and `npm run clean` delete the tracked `docs/ui-refactor/` documentation, including uncommitted edits.**
*File:* `webpack.config.js:10`, `package.json:7` (`clean` script).
*User-visible consequence:* Running either documented repository command silently destroys five tracked design documents and any uncommitted work in them. Encountered live during this review: `npm run build` deleted all of `docs/ui-refactor/`, including the uncommitted edits to `REFRACTORING_CONSTRAINTS.md`.
*Technical cause:* `output.path` is `docs/` with `clean: true`. Webpack's clean removes every file in the output directory not emitted by the build. The refactor branch introduced `docs/ui-refactor/` *inside* that output directory, so the branch's own documentation became build-sweepable. `npm run clean` was `rimraf docs`, deleting the directory outright.
*Evidence:* `git status` after `npm run build` reported `D docs/ui-refactor/BLOCK_COLOR_CLASSIFICATION.md`, `D … COMMAND_INVENTORY.md`, `D … DOM_DEPENDENCY_MAP.md`, `D … REFRACTORING_CONSTRAINTS.md`, `D … STYLESHEET_MAP.md`.
*Correction:* scope webpack's clean with `clean: { keep: /^ui-refactor[\\/]/ }`; scope the `clean` script to the emitted artifacts instead of the whole directory.
*Regression risk:* Low — stale non-`ui-refactor` output is still swept; verified `clean` → `build` reproduces a complete site.
*Tests required:* verified manually by `npm run build`, then `npm run clean && npm run build`, asserting five surviving files and byte-identical restored user edits.

### High

**H1 — Primary bottom-panel tabs have no accessible name at ≤620px.**
*File:* `src/index.html:307–309` (tabs), `src/assets/css/styles.css:1300` (`.viz-tab-label { display: none; }`).
*User-visible consequence:* On phone viewports, the unselected Problems / Output / Semantics tabs are announced as unnamed buttons by screen readers. Only the selected tab keeps a name. WCAG 2.1 SC 4.1.2 (Name, Role, Value) failure on the bottom panel's primary navigation.
*Technical cause:* The tabs' only text is `<span class="viz-tab-label">`, which the ≤620px rule hides for unselected tabs; the icon is `aria-hidden`, the count badge is `hidden`, and there was no `aria-label` or `title`.
*Evidence:* Browser probe at 390×844 with the bottom panel open:
`bottomTab-problems selected=true innerText="Problems"` / `bottomTab-output selected=false innerText=""` / `bottomTab-semantics selected=false innerText=""`, all with `aria-label=""`.
*Correction:* add `aria-label` (and matching `title`) to the three primary bottom tabs. The responsive icon-only design is preserved.
*Regression risk:* Very low — attribute-only; no layout or visual change.
*Tests required:* new — *primary bottom tabs keep accessible names where the phone layout hides their labels*.

**H2 — `#copyCode` produces an unhandled promise rejection and fails silently.**
*File:* `src/core/ui/layout.ts:409–422` (pre-fix).
*User-visible consequence:* When the clipboard write fails — denied permission, unfocused document, or an insecure (`http://`) context — the copy silently does nothing, the button still reads "Copy generated output", and the page raises an unhandled rejection. The user has no indication the copy failed.
*Technical cause:* `await navigator.clipboard.writeText(code)` in an `async` listener with no `try`/`catch`; the success state and revert timer sat after the unguarded `await`, so neither ran on failure.
*Evidence:* Browser probe stubbing a rejecting `navigator.clipboard`:
`UNHANDLED_REJECTIONS ["Error: Write permission denied."]`, `PAGE_ERRORS ["pageerror: Write permission denied."]`, `COPY_LABEL_AFTER_FAILURE Copy generated output`, `STATUS_AFTER_FAILURE Editing Lambda code.`
*Correction:* wrap in `try`/`catch`; on failure log and surface an `#icon-alert` state with "Could not copy — clipboard unavailable", reverting after 2400 ms.
*Regression risk:* Very low — the success path is unchanged.
*Tests required:* new — *a failing clipboard copy reports on the button instead of rejecting*, asserting the alert state, the revert, and zero page errors.

**H3 — Re-run and Arrange are unreachable at ≤620px.**
*File:* `src/assets/css/styles.css:1294–1295` (`.viz-tools #vizArrange, .viz-tools #vizRerun { display: none; }`), `src/core/ui/workbench.ts` palette array.
*User-visible consequence:* On phone viewports the only routes to "Re-run active semantic view" and "Arrange reduction steps" are hidden, and neither had a command-palette entry, keyboard shortcut, menu item, or any other surface. Both commands were completely unreachable — violating the contract that every command remain reachable through at least one interface, shortcut, or palette entry.
*Technical cause:* the responsive rule hides the buttons; the palette array covered 32 commands but omitted these two, as `COMMAND_INVENTORY.md` itself recorded ("Command palette: No").
*Evidence:* Browser probe at 390×844 with a semantic view active: `RERUN visible=false`, `ARRANGE visible=false`, `PALETTE rerun options=0`, `PALETTE arrange options=0`, `PALETTE total options=32`.
*Correction:* add `Run: Re-run Active Semantic View` and `Run: Arrange Reduction Steps` palette entries dispatching to the existing `#vizRerun` / `#vizArrange` handlers. Behavior is identical to the desktop buttons; only the route is new.
*Regression risk:* Very low — no handler changes; hidden elements still respond to `.click()`.
*Tests required:* new — *bottom-panel tools stay reachable through the palette where the phone layout hides them*.

### Medium

**M1 — `#themeToggle` is an invisible but focusable tab stop.** *(fixed)*
`src/index.html:265`. The theme state-holder is `.visually-hidden` yet keyboard-focusable, so Settings-dialog tab order contains an invisible checkbox that duplicates the visible segmented control. Probe: `{"focused":true,"tabIndex":0,"rects":1}`. Fixed with `tabindex="-1"` + `aria-hidden="true"` — the standard hidden-state-holder pattern, with the segmented control (which carries `aria-pressed`) remaining the accessible interface.

**M2 — Maximize and Restore render the identical glyph.** *(fixed)*
`src/index.html:189–190`. `.code-maximize-icon` and `.code-restore-icon` both referenced `#icon-fit`, so the CSS toggle at `styles.css:703–705` swapped one glyph for a visually identical one; the state was conveyed only by `title`/`aria-label`. Fixed by adding an `#icon-collapse` symbol (inward arrows) for the restore state. `aria-pressed` styling (`styles.css:265`) already changes both color and background, so state was never color-only.

**M3 — The accessibility test's unnamed-button check could not detect H1.** *(fixed)*
`tests/ui/workbench.accessibility.spec.ts:190`. The filter used `button.textContent`, which returns text regardless of `display: none`, so a control whose only label is hidden by a responsive rule still counted as named. This is precisely why H1 passed CI at 390×844. Switched to `button.innerText`, which reflects rendered text.

**M4 — `#showCodeFromWorkspace` double-fires on phone.** *(retained)*
`src/core/ui/layout.ts:356–366`. Both `click` and `pointerup` call `setCodeVisibility(true, true, true)`; `preventDefault()` on `pointerup` does not suppress the subsequent `click`, so on phone the panel is opened, persisted, scrolled, and focused twice. Effect is idempotent and not user-visible today. Not fixed: the dual binding looks like a deliberate workaround for a phone tap issue, and changing pointer semantics carries real regression risk on hardware not represented in this suite.

**M5 — Heading hierarchy skips a level.** *(retained)*
`src/index.html:224`. `#selectedTypeHeading` is an `<h3>` under the single visually-hidden `<h1>`, with no intervening `<h2>` in the main document flow (`<h2>`s exist only inside dialogs). Minor screen-reader outline inconsistency; changing heading levels risks visual regressions across the inspector and the visual baselines.

## 21. Known limitations

1. No formatter script exists; formatting was not verified.
2. Browser coverage is Chromium-only. Firefox and WebKit are unverified, notably for `<dialog>` focus behavior and pointer capture during resize.
3. Touch input is emulated by viewport size; no real touch-device pass.
4. Contrast ratios in `BLOCK_COLOR_CLASSIFICATION.md` are documented values, not computed by an automated test.
5. Accessibility checking is a hand-written contract, not axe-core; it covers names, roles, tab wiring, separators, landmarks, and duplicate IDs, but not full WCAG coverage.
6. M4 (phone double-fire) and M5 (heading skip) are open by decision, not oversight.
7. `docs/` remains both the published site and the home of `docs/ui-refactor/`; the C1 fix makes that safe but the coupling is inherently fragile.

## 22. Recommended future work

1. Move `docs/ui-refactor/` out of the webpack output directory entirely, removing the C1 coupling rather than working around it.
2. Add axe-core to the accessibility suite alongside the hand-written contract.
3. Add a computed contrast test so `BLOCK_COLOR_CLASSIFICATION.md` ratios are enforced rather than asserted in prose.
4. Add Firefox and WebKit projects to the Playwright config.
5. Introduce a single command registry so palette entries, menu items, and shortcuts derive from one source, making reachability holes like H3 structurally impossible.
6. Resolve M4 with a single pointer strategy, verified on real touch hardware.
7. Debounce the bottom-resizer persistence (L4) and refresh `aria-valuemax` on resize (L5).
8. Consider code-splitting to address the 900 KiB bundle warning.

## 23. Direction confirmation

The interface follows a quiet, domain-specific Block Lambda Calculus workbench direction.

Confirmed absent: gradients, glassmorphism, backdrop blur, decorative rounded cards (`--radius-panel: 0`), floating pills, glow effects, hero headings, KPI/dashboard cards, decorative badges, and excessive accent colors (one product accent). One coherent SVG sprite is the sole icon family. Font weights are limited to 500/600/700. Per-block arbitrary colors were replaced by seven muted grammatical families in which color is a secondary cue — connector geometry, labels, fields, inferred-type annotations, and diagnostics carry the fine-grained grammatical signal, and blocks are not flattened into one indistinguishable color.

Identity comes from the language: grammar-aware square connectors, the seven grammatical block families, generated lambda text, inferred types and typing derivations, the program outline, reduction traces, the CEK machine with environment and Kontinuation views, and the lockstep operational-correspondence view. Renderer internals stay out of normal chrome, the About dialog describes only implemented features and the real `.blc` format, and no MiniJava label or stale feature claim survives in production source.

The result reads as a visual lambda-calculus laboratory rather than a VS Code clone or a generic AI dashboard.
